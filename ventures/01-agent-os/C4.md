# C4 — Agent OS

## Nivel 1 — Contexto
Agent OS se ubica entre equipos que construyen agentes y los sistemas empresariales, proporcionando control operativo, políticas, observabilidad y memoria.

### Actores
- AI Engineer
- Platform Engineer
- Admin de seguridad
- Product Team
- Auditor interno / externo

### Sistemas externos
- LLM providers
- Bases de datos corporativas
- APIs empresariales
- Sistemas de identidad
- Herramientas de observabilidad
- Repositorios y CI/CD

## Nivel 2 — Contenedores
1. **Admin Console**
   - UI para registro, monitoreo y gobierno de agentes.
2. **API Gateway**
   - Entrada única para UI, SDKs y automatizaciones.
3. **Agent Runtime Service**
   - Ejecuta agentes y coordina tools, memoria y políticas.
4. **Scheduler Service**
   - Lanza tareas programadas o por evento.
5. **Policy Engine**
   - Evalúa permisos, límites y reglas.
6. **Memory Service**
   - Provee memoria semántica y de estado.
7. **Audit Service**
   - Registra eventos, replays y trazabilidad.
8. **Tool Adapter Layer**
   - Integra tools internas y externas.
9. **Metrics & Alerting Layer**
   - Métricas, trazas, dashboards y alertas.
10. **Billing & Quotas Service**
    - Mide uso y limita ejecución.

## Nivel 3 — Componentes clave del Agent Runtime
- Execution Orchestrator
- Context Builder
- Tool Invocation Manager
- Step Recorder
- Failure Handler
- Result Publisher

## Nivel 3 — Componentes clave del Policy Engine
- Auth Context Resolver
- Resource Policy Evaluator
- Rate Limit Evaluator
- Risk Gate
- Approval Hook

## Nivel 3 — Componentes clave de Memory Service
- Conversation Store
- Semantic Recall
- State Snapshot Store
- Retention Manager

## Flujo crítico
1. Un usuario o sistema dispara una tarea.
2. API Gateway valida identidad.
3. Scheduler o Runtime crea ejecución.
4. Policy Engine autoriza.
5. Runtime arma contexto y consulta memoria.
6. Tool Adapter ejecuta llamadas externas.
7. Audit Service registra cada paso.
8. Metrics layer emite telemetría.
9. Resultado queda disponible en consola o API.

## Decisiones arquitectónicas
- Multi-tenant desde el inicio.
- Modelo-neutral y framework-neutral.
- Separación estricta entre runtime, memoria y políticas.
- Auditoría inmutable como capacidad central, no adicional.
